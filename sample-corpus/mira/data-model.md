# Data Model

## The Mira Data Model

Everything stored in Mira is organized into a hierarchy: account → collections → namespaces → vectors. Understanding how these layers relate to each other is important for designing an efficient schema.

An account is the top-level container. All collections, API keys, roles, and billing belong to a single account. Collections are the primary organizational unit within an account. A collection groups vectors that share the same number of dimensions and the same similarity metric. You cannot search across multiple collections in a single query, so collections should be defined by data type or search domain, not by operational concerns like environment or tenant.

Within a collection, namespaces provide logical isolation. A namespace is a string label attached to each vector at insert time. Searching within a namespace only considers vectors that carry that label, even though all namespaces share the same underlying index. This makes namespaces cheap: there is no storage or indexing overhead to having many namespaces in one collection. They are well-suited for multi-tenancy, A/B testing, or separating live data from staging data within a single index.

A vector is the atomic unit of data. Each vector consists of four parts: an ID, a values array, optional metadata, and an optional namespace. The ID is a string chosen by the caller. IDs must be unique within a namespace; two vectors in different namespaces can share the same ID without conflict. The values array is the embedding—a list of floating-point numbers whose length must exactly match the collection's configured dimensions. If the length does not match, the insert is rejected with a `DimensionMismatchError`.

Metadata is a flat key-value map attached to a vector. Keys are strings. Values must be strings, numbers, or booleans; nested objects and arrays are not supported. Metadata is stored separately from the vector index and is retrieved as part of search results. It is also indexed for filtering, which allows search queries to constrain results to vectors where specific metadata fields match specific values. The indexing is done at insert time, so changing metadata (via `update()`) does not require re-embedding the vector.

The relationship between IDs and upsert semantics is important to understand. When you call `insert()` with a record whose ID already exists in the same namespace, Mira replaces the entire record: both the vector values and the metadata are overwritten. This is unconditional. If you want to change only metadata without touching the embedding, use `update()`. If you want to change only the embedding without touching the metadata, you must read the existing metadata first and pass it along with the new vector to `insert()`. There is no partial-update operation for vector values.

Deletion in Mira is by ID or by metadata filter. Deleting by ID removes a specific vector. Deleting by filter removes all vectors matching the filter criteria, which can be a large number of vectors. Neither operation is transactional with inserts: if a delete and an insert on the same ID arrive concurrently, the result is undefined. For bulk operations that require consistency, use namespaces to implement a blue/green pattern: insert into an inactive namespace, then atomically switch queries to the new namespace once the insert is complete, and delete the old namespace afterward.

Vectors are immutable in the sense that their dimensionality is fixed at collection creation time and cannot change. This is a fundamental constraint of ANN indexes: the index geometry is built for a specific number of dimensions, and changing it would require rebuilding the entire index from scratch. If you need to migrate from one embedding model to another (different dimensions), you must create a new collection, re-embed all your content, and insert into the new collection. The old collection can then be deleted. There is no in-place migration path.

The Mira data model does not have a concept of documents, chunks, or any higher-level structure above the vector. If you are building a retrieval system where each document is split into multiple chunks and each chunk is a separate vector, the relationship between chunks and their parent document is your responsibility to encode in metadata. A common convention is to store the source document path or ID as a metadata field (e.g., `{ document_id: 'readme.md', chunk_index: 3 }`) so that search results can be grouped or deduplicated at the application layer. Mira does not enforce or interpret these fields; they are transparent metadata.

Storage billing is based on the number of vectors and their dimension count. A collection of 1 million 384-dimensional vectors uses roughly 1.5 GB of storage for the raw embeddings, plus additional storage for the HNSW graph (~30% overhead), FTS index if enabled, and metadata. Metadata storage depends on value sizes; a typical key-value pair of 20 bytes costs roughly 20 MB per million vectors. Plan your metadata schema to be as concise as possible if you are storing very large numbers of vectors.
